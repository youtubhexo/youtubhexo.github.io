端到端侧式

发表于 2024-03-30 本文字数： 657 阅读时长 ≈ 2 分钟

端到端（End-to-End，简称 E2E）测试是一种测试方法，旨在模拟用户真实的操作场景，从应用程序的开始到结束，验证整个系统是否按照预期运行。这种测试通常涉及多个组件、服务和系统，以确保它们在集成时能够正确地工作。

以下是关于端到端测试的具体介绍：

  1. 测试范围： E2E 测试覆盖了整个应用程序的功能。它模拟了用户与应用程序进行交互的过程，从用户输入数据、点击按钮，到查看结果或响应。这样的测试可以验证整个应用程序的工作流程是否正确。

  2. 自动化执行： 虽然端到端测试覆盖了整个应用程序的功能，但它们通常是自动化执行的。开发团队使用自动化测试框架和工具来编写和运行这些测试，以确保测试过程的可重复性和效率。

  3. 模拟用户行为： E2E 测试旨在模拟用户真实的操作行为。这意味着测试将会模拟用户在应用程序中进行的各种操作，比如输入文本、点击按钮、导航页面等，以验证应用程序在各种交互条件下的行为是否正确。

  4. 测试环境： 为了执行端到端测试，通常需要搭建一个与生产环境相似的测试环境。这包括部署应用程序到测试服务器上，并可能需要模拟真实数据、用户、网络环境等条件。

  5. 验证预期结果： 端到端测试通常会定义一系列预期结果。测试将模拟用户操作，并验证应用程序的行为是否符合预期。如果发现任何与预期不符的情况，测试将会失败，并且可以帮助开发团队迅速定位和修复问题。

  6. 跨平台兼容性： 端到端测试需要确保应用程序在不同的平台、浏览器和设备上都能够正常运行。因此，测试通常会在多个不同的平台和浏览器上执行，以验证应用程序的跨平台兼容性。

总的来说，端到端测试是一种全面的测试方法，用于验证整个应用程序的功能和行为。通过模拟用户真实的操作场景，开发团队可以确保应用程序在各种条件下都能够按照预期工作。

集成测试理解

发表于 2024-03-30 本文字数： 468 阅读时长 ≈ 2 分钟

在开发 Vite 项目过程中，集成了许多第三方插件后，验证插件的过程可以被归类为集成测试的一部分。集成测试的目标之一就是验证不同组件、模块或插件在集成后是否能够正常工作。因此，对于已经集成到项目中的第三方插件，验证其与项目的集成情况以及与其他组件之间的交互是否符合预期，可以被视为集成测试的一部分。

在这种情况下，验证第三方插件可能涉及以下方面：

接口测试： 确保项目与第三方插件之间的接口和集成是正确的，比如确保正确引入插件、正确配置插件等。

功能性测试： 验证项目在集成了第三方插件后的功能是否仍然正常。这可能涉及到使用插件提供的功能，以确保它们按预期工作。

性能测试： 确认项目在集成了第三方插件后的性能是否受到影响。有些插件可能会对项目的性能产生影响，因此需要验证这些插件的使用是否符合性能要求。

兼容性测试： 确保项目在集成了第三方插件后仍然能够在不同的环境和平台上正常运行。这可能涉及到验证项目在不同浏览器、设备上的兼容性。

通过对集成了第三方插件的项目进行测试，开发团队可以确保这些插件与项目的集成是正确的，并且项目在集成了这些插件后仍然能够正常工作。这有助于提高项目的质量和可靠性，减少在生产环境中出现问题的风险。

Vue 3 常用函数和类型的整理

发表于 2024-03-01 本文字数： 468 阅读时长 ≈ 2 分钟

当然，下面是这个 TypeScript 声明文件中包含的一些 Vue 3 常用函数和类型的整理：

函数：

  • computed: 创建一个计算属性。
  • createApp: 创建一个 Vue 应用实例。
  • customRef: 创建一个自定义的 ref。
  • defineAsyncComponent: 定义一个异步组件。
  • defineComponent: 定义一个 Vue 组件。
  • nextTick: 在下次 DOM 更新循环结束之后执行延迟回调。
  • onActivated: 组件激活时的钩子函数。
  • onBeforeMount: 组件挂载之前的钩子函数。
  • onBeforeUnmount: 组件卸载之前的钩子函数。
  • onBeforeUpdate: 组件更新之前的钩子函数。
  • onDeactivated: 组件停用时的钩子函数。
  • onErrorCaptured: 捕获组件渲染错误的钩子函数。
  • onMounted: 组件挂载后的钩子函数。
  • onRenderTracked: 跟踪组件渲染的钩子函数。
  • onRenderTriggered: 渲染触发的钩子函数。
  • onScopeDispose: 作用域销毁时的钩子函数。
  • onServerPrefetch: 服务器端预取的钩子函数。
  • onUnmounted: 组件卸载后的钩子函数。
  • onUpdated: 组件更新后的钩子函数。
  • provide: 提供属性注入的方法。
  • reactive: 将对象转化为响应式对象。
  • readonly: 将对象转化为只读的响应式对象。
  • ref: 创建一个 ref。
  • shallowReactive: 创建一个浅响应式对象。
  • shallowReadonly: 创建一个只读的浅响应式对象。
  • shallowRef: 创建一个浅的 ref。
  • toRaw: 获取代理对象的原始对象。
  • toRef: 将一个响应式对象转换为 ref。
  • toRefs: 将一个响应式对象的所有属性都转换为 ref。
  • triggerRef: 触发一个 ref。
  • unref: 取消一个 ref。

类型：

  • Component: Vue 组件类型。
  • ComponentPublicInstance: Vue 组件实例类型。
  • ComputedRef: 计算属性引用类型。
  • InjectionKey: 注入键类型。
  • PropType: 属性类型。
  • Ref: ref 类型。
  • VNode: Vue 虚拟节点类型。
  • WritableComputedRef: 可写计算属性引用类型。

探索自我成长的旅程：道与术的认知之路

发表于 2024-02-29 本文字数： 628 阅读时长 ≈ 2 分钟

在人生的旅途中，我们时常迷失在各种挑战和困惑之中。但在这漫长的道路上，有两个关键元素——道与术，它们引导着我们不断认知自我，迈向成长之路。

道，是人生的意义和方向。它不仅是对生命深层次的思考，更是一种生活态度。道教导我们如何面对挫折和困境，坚持信念和勇气。通过追寻道，我们找到了自己的位置和价值，感受到生命的深刻意义，更坚定地踏上人生之路。

术，则是实现道的方法和技巧。它是我们在实践中采取的行动和策略，是达成目标的手段。通过学习和实践术，我们能够更好地应对挑战，解决问题，实现理想和目标。术让我们不断成长进步，高效地生活和工作，更深刻地理解和实践道。

在我认识到这一点后，我开始探索自我成长的旅程。我看到了各种精英是如何处理事务的，然后融入了社会实践。当我终于赚了一笔钱，实现了一个初步的闭环时，我开始思考自己的认知水平。

在认知的低层次中，我陷入了对事情带来的直接收益的追逐中，不断投入，直到事情崩溃。但在认知的高层次，我开始问自己更深层次的问题：我想成为什么样的人？通过什么方式去实现？环境是否允许？我是否有足够的内在动力来战胜环境的限制？是否关注行业动态？是否了解技术潮流？

然而，在初步闭环中，我常常遇到两个问题：一是无法完全把握环境中的某些因素；二是处在认知的较低层次中。

为了解决这些问题，我不断思考，消化已有的知识和经验，勇于探索未知的领域。通过思考和实践，我不断地成长，逐渐提升认知水平，更清晰地认识自我，更坚定地走向成功和幸福的人生之路。

因此，在这个探索自我成长的旅程中，让我们始终牢记道与术的重要性，不断追寻道，不断学习和实践术，从而更加清晰地认识自我，更加坚定地走好自己的人生之路。

本该相同的事情，却因人而异

发表于 2024-02-27 本文字数： 709 阅读时长 ≈ 3 分钟

相同的事情，不同的人结果却不同。

我并不享受被本该直觉性工作的工具打断我的工作流程。 沉默的文字，迸发出无穷的力量。

正如 vite 所宣称的按需加载，优化开发体验。

在当今数字化的时代，我们常常面对着相同的工具、相同的任务，但是结果却因为个人的偏好、习惯或者工作流程而大相径庭。在这篇博文中，我将探讨这种现象，并分享我个人对于工具与工作流程的看法。

在我看来，我并不享受被本该直觉性工作的工具打断我的工作流程。对我而言，沉默的文字才是最能够迸发出无穷力量的工具。文字无声地等待着我们的引导，我们可以在其中找到灵感，表达思想，甚至改变世界。相比之下，过度复杂的工具往往会破坏我的专注力，打断我的创作思路，让我远离本应该专注于的创造性工作。

然而，正如人们常说的一句话：“同一件事情，不同的人会有不同的看法。”就像使用开发工具这样的事情，每个人都有自己的喜好和习惯。有些人可能会喜欢一站式的开发环境，希望通过集成的工具来提高效率，而有些人则更喜欢自定义工作流程，选择适合自己需求的工具进行开发。正是因为这种差异，才使得同一个任务在不同人手中的完成方式截然不同。

就像最近流行的开发工具 Vite 所宣称的那样，它提供了按需加载的功能，优化了开发体验。这种灵活性使得开发者可以根据自己项目的需求，灵活调整加载方式，提高开发效率。然而，对于某些开发者来说，这种新功能可能并不是他们所需要的，他们更倾向于使用传统的加载方式，或者其他更符合自己习惯的工具。

总之，即使面对着相同的任务和工具，我们每个人都有自己独特的方式去完成它们。这种个性化的差异是多样性的体现，而不是需要被统一或者排斥的因素。因此，在选择工具和制定工作流程时，我们应该充分尊重个人的喜好和习惯，而不是盲目跟随潮流或者他人的选择。

通过理解并尊重这种差异，我们可以更好地发挥个人的潜力，提高工作效率，创造更加丰富多样的成果。
